- gate G5: PASS · L4 fresh-context review approved; quiz-me Q&A answered + logged.
- QUIZ-ME (user answers, logged verbatim):
  - Q1 (design) — "We pin the development toolchain (ruff, mypy, pytest, etc.) because they are part of
    the verification system, not the product itself. Unbounded tool versions can introduce new rules,
    diagnostics, or behavior that cause previously verified code to fail without any source changes.
    Pinning them makes the CI pipeline deterministic and reproducible... Runtime dependencies follow
    their own compatibility strategy, but the verification environment must remain stable."
  - Q2 (edge) — "No. A statement-only coverage metric does not detect branch-only gaps. If new conditional
    paths are added without reducing statement coverage, the domain percentage can still remain at 100%.
    Statement coverage guarantees line execution, not branch completeness. Detecting branch-only gaps
    would require branch coverage metrics or dedicated tests beyond the current statement-weighted gate."
  - Q3 (change-impact) — "After the M8 change, APP_NAME represents only the application name, while
    APP_USER_NAME represents the approver identity. If a user already has both variables defined, the
    application continues to work without migration because each setting now maps to a distinct purpose.
    Existing configurations using only APP_NAME for the approver would need to add APP_USER_NAME, but
    configurations already containing both variables require no changes."
- decision: M8 VERIFIED. implementation-notes.html + CURRENT.md synced. Wiki seeding deferred to separate post-M8 task (per user directive). DONE.html/PLAN.md remain locked/untouched.
