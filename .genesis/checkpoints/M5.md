## Goal
Deliver an independently-demoable Gmail OAuth authentication capability: build the
consent URL (PKCE public client), run the complete interactive flow (browser launch
+ local loopback callback + authorization-code capture), exchange the code for
tokens, refresh expired access tokens, and persist tokens securely on the local
machine. M5 ends with valid stored OAuth credentials. NO email delivery / Gmail
send exists in M5 — delivery is exclusively M6, and it begins from these stored
credentials.

## Architecture decisions (approved 2026-07-17)
- D1 (Libraries): use `google-auth-oauthlib` + `google-auth` for OAuth.
- D2 (Client type): PKCE public client — NO client secret sent or stored.
- D3 (Token storage): tokens stored OUTSIDE the repo at a caller-supplied path
      (e.g. oauth_tokens.json), written with 0600 permissions, gitignored.
- D4 (Scope): request only `https://www.googleapis.com/auth/gmail.send`
      (single consent covering the M6 send at M5 auth time).
- D5/D7 (Complete flow in M5): `GoogleOAuthClient.authenticate()` runs the FULL
      interactive flow (browser + loopback + code capture) and ends with valid
      stored credentials; M6 begins from those credentials and implements delivery
      only. No DeliveryService / send method exists in M5.
- D6 (Settings): `GmailSettings` added as a section of the existing `Settings`
      hierarchy (GMAIL_ env prefix, lazy default_factory) — mirrors the M3 ADR-1
      config pattern; no standalone config module.

## Extra engineering decisions (locked, freeze constraints for M6)
- X1: `OAuthTokenStore` exposes a convenience `has_valid_tokens()` API.
- X2: a dedicated `GoogleOAuthClient` owns ALL OAuth operations (URL/PKCE build,
      code exchange, refresh, refresh_if_needed, full interactive authenticate).
      Gmail *delivery* remains a separate concern introduced in M6 — no
      `gmail_adapter.py` send surface in M5.
- X3: Authentication completes before Delivery exists — M5 ships no
      DeliveryService/send path (structural guarantee, enforced by test).

## M5 → M6 contract (carry-forward)
- M6 DeliveryService MUST call `GoogleOAuthClient.refresh_if_needed()` before every
  Gmail API request: transparent refresh on mere expiry (no interactive block).
- If `refresh_if_needed()` / `refresh_access_token()` raises `TokenRefreshError`
  (e.g. revoked refresh token), the store is cleared. M6 must HARD STOP: surface the
  auth failure, require re-authentication, and only then retry delivery of the SAME
  immutable `EmailMessage`. Never send with a stale/expired token; never silently
  retry; never rebuild the message from the draft (preserves M4-E4 artifact_identity).

## L4 quiz-me (human answers recorded)
- Q1 (design): A desktop app is a public client — the binary is under user control,
  so any embedded client secret can be reverse-engineered and cannot be confidential.
  PKCE removes the need to trust a secret. An extracted client secret does NOT grant
  live access to the user's Gmail or tokens; at most it lets an attacker impersonate
  the application's IDENTITY during the OAuth flow. Real access protection is the
  user's authentication + consent + issued tokens, which are separate from the
  client identity.
- Q2 (edge): Correct/safe. On a revoked refresh token, refresh_access_token raises
  TokenRefreshError and OAuthTokenStore clears stored creds to prevent reuse. The
  previously-created EmailMessage remains a valid immutable business artifact but
  must NOT be delivered until auth is re-established. M6 treats a cleared store as a
  hard stop: re-authenticate, obtain fresh creds, THEN retry delivery of the existing
  EmailMessage. Never silently retry with stale creds; never rebuild the message.
- Q3 (impact): M6 DeliveryService calls refresh_if_needed() before every Gmail
  request. Merely-expired token → refreshed transparently, delivery proceeds without
  interactive re-auth. If refresh_if_needed() raises TokenRefreshError, M6 stops
  immediately, surfaces the auth failure, requests re-authentication, does NOT deliver
  with an expired token, does NOT silently retry, and preserves the existing
  EmailMessage for unchanged delivery after re-auth. Preserves outbound_timeout and
  artifact_identity.
