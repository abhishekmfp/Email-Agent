## Architecture decisions (approved 2026-07-17)
Deliver the immutable, approved `EmailMessage` through the Gmail API (OAuth)
via a thin `DeliveryService` + `GmailAdapter`. Explicit timeout, no auto-resend
on uncertain delivery. Real Gmail send. NO drafting, NO OAuth logic, NO
approval logic — those remain M3/M5/M4 responsibilities respectively.

## Architecture decisions (approved 2026-07-17)
- D1 (Dependency): add `google-api-python-client>=2.140` as the Gmail API
      client. OAuth itself stays in M5 (`google-auth-oauthlib`); M6 only calls
      `GoogleOAuthClient.refresh_if_needed()` before each send.
- D2 (MIME location): MIME/RFC822 construction stays INSIDE `GmailAdapter`
      (`_build_raw`), deliberately NOT extracted to a `MimeMessageBuilder`.
      Rationale: V1 has one provider + one message shape; a separate builder
      would be a speculative abstraction with no second caller, violating the
      project's "extract abstractions only after multiple concrete
      implementations" rule. `_build_raw` is a small pure method — the obvious
      future extraction point if a 2nd provider/multipart need arises.
- D3 (SENT ownership — D-M6-1 Option a): `DeliveryService.send()` returns a
      `DeliveryResult` (message_id/thread_id/status). The CALLER (interface /
      `ApproveEmailUseCase`) owns the SENT lifecycle transition. `DeliveryService`
      is transport-only and never mutates the frozen `EmailMessage`.

## Extra engineering decisions (locked, freeze constraints for M7+)
- X1 (artifact_identity): `DeliveryService` hands the EXACT `EmailMessage`
      instance to `GmailAdapter.send()`; the adapter reads it read-only and
      NEVER rebuilds from a draft. `test_send_approved_message_delivers` asserts
      `adapter.send.call_args.args[0] is message`.
- X2 (refresh before every send): `DeliveryService.send()` calls
      `GoogleOAuthClient.refresh_if_needed()` before EVERY Gmail request
      (M5→M6 contract). Merely-expired token → refreshed transparently; delivery
      proceeds without interactive re-auth.
- X3 (TokenRefreshError = hard stop): if `refresh_if_needed()` raises
      `TokenRefreshError` (e.g. revoked refresh), `DeliveryService` returns
      `DeliveryResult(status="auth_failed")` and NEVER calls `GmailAdapter.send`.
      M5 already clears the store; M7+ must surface re-auth and retry the SAME
      immutable `EmailMessage`. Never deliver on a stale token; never silently
      retry; never rebuild from draft.
- X4 (retry = pre-dispatch only): bounded exponential backoff
      (`_RetryPolicy`, default 3 attempts) applies ONLY to
      `PreDispatchTransportError` (connection refused / timeout before Gmail
      acknowledges). Once a send is dispatched, it is NEVER retried.
      `UnknownDeliveryStateError` (dispatched but outcome unconfirmed) is
      reported immediately and NOT retried (decisions-manifest partial-failure
      rule: "never auto-resend when uncertain").
- X5 (timeout): `GmailSettings.send_timeout_seconds` (default 30s) is passed to
      the Gmail `users.messages.send` call (outbound_timeout invariant). Token
      refresh timeout remains covered by M5's `request_timeout_seconds`.
