## L4 quiz-me (human answers recorded)
- Q1 (design decision): PromptBuilder extracted to separate prompt engineering from
  transport logic; AnthropicAdapter only talks to the API, PromptBuilder owns prompt
  construction → prompts evolvable/testable/versionable independently. Typed
  DraftResponse (not raw dicts) is the explicit AI↔app contract → type safety,
  explicit validation, fewer runtime errors, no unstructured LLM output propagates.
- Q2 (edge case): clarification_required=false + empty body → DraftResponseValidator
  raises DraftResponseValidationError; DraftingService catches it and returns an
  ERROR DraftResult; EmailDraft is never created. llm_output_validated + state_ownership
  invariants preserved.
- Q3 (change impact): AnthropicSettings added via default_factory is additive —
  existing get_settings() users keep working unchanged; only the returned object's
  shape gains an `anthropic` section. Backward-compatible, non-breaking.

## Locked M3 ADRs (do not re-litigate in M4–M8)
1. AnthropicSettings lives in config/settings.py (existing config system), NOT a new
   standalone module.
2. anthropic dependency pinned (==0.116.0) in pyproject.toml + uv.lock regenerated.
3. AnthropicAdapter returns a typed DraftResponse; never exposes raw dicts. DraftResponse
   is validated BEFORE EmailDraft creation (llm_output_validated).
4. The boundary validator is named DraftResponseValidator (renamed from the planned
   DraftValidator).
5. Retry policy lives in DraftingService, NOT AnthropicAdapter. AnthropicAdapter performs
   exactly ONE request per call.
6. Prompt construction extracted into a dedicated PromptBuilder, NOT embedded in
   AnthropicAdapter.
7. The Domain layer stays completely unaware an LLM exists (domain_inward invariant).

## Risks that materialized & mitigations
- Pydantic enforces DraftResponse field types at construction, so the validator's
  original isinstance type-guards were dead code → trimmed to semantic/presence checks
  only; type conformance covered by DraftResponse tests.
- DraftingService must catch DraftResponseValidationError (not just AnthropicProviderError)
  and surface an ERROR DraftResult, not crash the workflow.
- Anthropic APIError signature is SDK-version-specific → tests use a signature-agnostic
  subclass stand-in rather than constructing APIError directly.
