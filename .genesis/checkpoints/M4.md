## Goal
Convert an AWAITING_APPROVAL EmailDraft into an immutable EmailMessage through
explicit human approval. Approve / Edit / Reject flow. NO Gmail/send in M4
(delivery is M6). Reject/edit invalidates prior approval; only an explicitly
re-approved EmailMessage is deliverable.

## Key realization (scope)
The entire domain side of M4 was already shipped in M2:
- `EmailDraft.approve(Approval)` — guards state, returns APPROVED copy
- `EmailDraft.edit(...)` — returns copy in AWAITING_APPROVAL, wipes approval
- `EmailDraft.to_message()` — builds frozen EmailMessage (guarded by can_deliver)
- `EmailMessage` — frozen dataclass (dataclass(frozen=True))
- `Approval` — domain value object (approver + decided_at)
- `ApprovalPolicy.can_deliver` — object-identity approval check (M2 ADR)

So M4 is **application orchestration + tests only**. No domain file was modified.

## Architecture decisions (approved 2026-07-16)
- D1 (Regenerate owner): Option A — regenerate handled by DraftEmailUseCase
      (interface re-invokes it); ApproveEmailUseCase stays LLM-free.
- D2 (Name): keep `ApproveEmailUseCase` (matches locked context-graph/PLAN node).
- D3 (Preview): render directly from EmailDraft; no Preview model introduced.
- D4 (Command shape): V1 single ApprovalDecision dataclass (optional edit fields);
      evolution to explicit command objects is a future ADR candidate if command
      complexity grows.

## Extra engineering decisions (locked, freeze constraints for M6)
- E1: ONLY `ApproveEmailUseCase` may construct the domain `Approval` object.
      No interface/other app code builds Approval. This keeps approval a business
      operation owned by the application layer and preserves the trust boundary.
- E2: `ApproveEmailUseCase` is LLM-free; it never imports AnthropicAdapter,
      DraftingService, or the LLM SDK.
- E3: EmailMessage is created exactly ONCE, at APPROVE, inside
      ApproveEmailUseCase via `approved.to_message()`; it is frozen; later edits to
      the original draft MUST NOT change it.
- E4 (artifact_identity): M6 DeliveryService must deliver the exact EmailMessage
      instance produced by ApproveEmailUseCase — it must NEVER recreate/derive an
      EmailMessage from the (possibly edited) draft. DeliveryService is transport
      only (read-only over the EmailMessage).

## Invariants preserved (context-graph)
- domain_inward: domain imports no infrastructure/framework (unchanged).
- approval_gate: no email reaches Gmail in M4 (ApproveEmailUseCase never invokes
  GmailAdapter; it stops after returning EmailMessage).
- artifact_identity: EmailMessage == approved draft content, frozen at approve,
  immutable after later edits.
- state_ownership: workflow state lives in domain (DraftState + guarded
  transitions); LLM owns none.

## L4 quiz-me (human answers recorded)
- Q1 (design): Only ApproveEmailUseCase builds Approval because approval is a
  business operation, not UI. Centralizing it ensures every approval follows the
  same validation/workflow rules and preserves the trust boundary; if the
  interface built Approval directly, interfaces could bypass app policies and
  create inconsistent approval — violating the invariant that only the app layer
  authorizes AWAITING_APPROVAL -> APPROVED.
- Q2 (edge): Editing an APPROVED draft invalidates the existing approval and resets
  to AWAITING_APPROVAL. The old EmailMessage stays immutable as a historical
  artifact but is no longer the deliverable; the edited draft needs a fresh
  Approval -> new EmailMessage. Preserves Approved Artifact == Delivered Artifact.
- Q3 (impact): ApproveEmailUseCase returns the immutable EmailMessage and stops.
  M6 DeliveryService must treat it as read-only and deliver it exactly as received,
  never recreating from EmailDraft — transport only — or artifact_identity breaks.
