---

## 1 · Objective

Expose the already-VERIFIED core (M1–M6: draft → validate → approve → immutable
`EmailMessage` → Gmail delivery) through two thin, deterministic outer surfaces —
a **FastAPI** HTTP interface and a **CLI** — plus the minimum cross-cutting
observability the spec requires: `GET /health`, structured logging with secret
masking, and a unified structured error envelope. The interface layer is a
**delegating shell**: it calls the existing application use cases and never owns
business rules, approval logic, or delivery internals.

Non-goal: M7 adds no new capability to the cognitive/delivery pipeline. It makes
the existing pipeline reachable by a human over HTTP and CLI.

## 2 · Success Criteria

- `GET /health` returns `{"status":"healthy","version":"<ver>"}`; exposes **no**
  sensitive config (no keys, tokens, recipients, prompts, SMTP/OAuth details).
- Full flow works over HTTP **and** CLI with mocked Gmail: draft → approve → send,
  and the edited/reject branches behave correctly (edit invalidates prior approval;
  reject sends nothing).
- Every external/infra failure (LLM error, OAuth/TokenRefreshError, Gmail transport
  / UnknownDeliveryState) is surfaced as a **structured error response** (typed
  envelope), never as an unhandled 500 / traceback leak.
- Structured logging emits on every significant transition (startup/shutdown,
  request received, draft created, approval granted/denied, delivery dispatched,
  delivery result, retries, errors) and **masks** recipient addresses — never logs
  email bodies, API keys, OAuth tokens, auth headers, or prompt contents.
- Gates green: `uv run ruff check src/ tests/ && uv run mypy src/email_agent &&
  uv run pytest -q`.
- Coverage: Application ≥ 90%, Overall ≥ 80% (interface layer well-covered; inner
  layers unchanged so their existing coverage is preserved).
- No `context-graph.json` invariant violated by the change (verified by L4).

## 3 · Scope (IN)

- **FastAPI interface** — `src/email_agent/interface/api/*.py`: app factory, routers,
  dependency injection wiring, `GET /health`, request/response models, error handlers.
- **CLI interface** — `src/email_agent/interface/cli/*.py`: local entry that drives
  the same use cases (draft / approve / edit / reject / send verbs).
- **Request/Response models** — typed pydantic (or dataclass) DTOs that translate
  between the wire and the domain/application contracts. These are **transport models
  only**; they must not duplicate or replace domain value objects.
- **Dependency wiring** — compose `DraftEmailUseCase` + `ApproveEmailUseCase` +
  their dependencies (DraftingService, DeliveryService, adapters, OAuth client,
  token store, settings) and inject them into the interfaces. `main.py` wiring updated.
- **`GET /health`** — version + status only.
- **Structured logging** — centralized structured logger (console-structured now,
  JSON-easy later per manifest); consistent secret-safe formatting; recipient masking
  helper reused everywhere.
- **Secret masking** — a single masking utility applied to logs AND error responses;
  recipients shown masked (e.g. `a***@company.com`), bodies/keys/tokens/prompts never
  rendered.
- **Error responses** — unified structured error envelope across HTTP + CLI; maps
  domain/application/infra failures to safe, typed responses.

## 5 · Architectural Constraints (binding — carries M3–M6 freeze forward)

### 5.1 Context-graph invariants (L4-enforced, must hold)
- **domain_inward** — interface layer imports no domain business rules; it depends on
  application use cases only. Domain still imports nothing from infra/framework.
- **approval_gate** — every send path goes through `ApproveEmailUseCase` (valid
  `Approval` in APPROVED). The interface MUST NOT expose a send endpoint that calls
  `DeliveryService` directly or bypasses approval.
- **llm_output_validated** — unchanged; interface does not alter the M3 validation path.
- **outbound_timeout** — unchanged; M7 adds no new outbound network calls.
- **secrets_hygiene** — logs/errors/`/health` never expose keys, tokens, recipients
  (full), bodies, or prompts. Withhold-from-LLM already satisfied by inner layers.
- **artifact_identity** — the interface delivers the **exact** `EmailMessage` instance
  `ApproveEmailUseCase` produced; it never rebuilds one from a draft.
- **state_ownership** — application owns all workflow state; interface owns none.

### 5.2 Locked decisions M7 must honor (freeze constraints)
- **M4-E1**: ONLY `ApproveEmailUseCase` constructs domain `Approval`; the interface
  never builds `Approval`.
- **M4-E2**: `ApproveEmailUseCase` is LLM-free; interface must not import
  `AnthropicAdapter`/`DraftingService`/LLM SDK.
- **M4-E3/E4 + M6-X1**: `EmailMessage` is created once at approve inside
  `ApproveEmailUseCase` and is frozen; `DeliveryService` delivers that exact instance.
  Interface triggers send only via `ApproveEmailUseCase`.
- **M6-D3/X1**: the CALLER owns the SENT transition — interface/use case flips
  `EmailMessage` → SENT from the returned `DeliveryResult`; `DeliveryService` stays
  transport-only.
- **M6-X2/X3**: `DeliveryService.send()` already calls `refresh_if_needed()` before
  every Gmail request; on `TokenRefreshError` it returns `DeliveryResult(auth_failed)`
  and never calls the adapter. Interface must surface `auth_failed` as a structured
  re-auth-prompt error, retrying the SAME immutable `EmailMessage` (never rebuild).
- **M6-X4**: `UnknownDeliveryStateError` is reported, never auto-resent — interface
  surfaces it as a structured "verify delivery" error.
- **M3-ADR7**: domain stays LLM-agnostic; interface must not leak LLM SDK into domain
  (it won't — interface depends on application only).
- **M2 evo (non-blocking, note only)**: RFC validation in infra; approval identity may
  later become persistent; `decided_at` may become mandatory on audit — M7 does not
  pre-empt these.

### 5.3 Wiring rule
Interface modules import **only** from `application` (use cases) + `config` +
`infrastructure` *types they are given via DI*. They must NOT reach into
`domain` rules or `infrastructure` adapter internals except through the injected use
cases / settings. No new side effects beyond those the use cases already perform.

### 5.4 One-use-case-per-endpoint rule (approved 2026-07-17, added post-charter)
- **AC-UI-1:** Every interface endpoint — each HTTP route and each CLI verb — MUST
  delegate to **exactly one** Application use case. The interface is a pure translation
  shell: request → single use-case call → response.
- The interface MUST NOT duplicate orchestration or business logic. It does not compose
  multiple use cases, re-implement approval/validation/delivery rules, or hold workflow
  state.
- Multi-step flows (draft → approve → send) are expressed by the **client** issuing
  separate endpoint calls in sequence; the interface does not chain use cases internally.
  (This resolves open Q6: the `/send` endpoint calls `ApproveEmailUseCase` exactly once —
  which internally triggers `DeliveryService` — and the interface composes nothing.)

## 7 · Risks & Mitigations

- **R1 Interface layer leaking business rules** (domain_inward violation).
  Mitigation: interface delegates to use cases only; add a scope test asserting no
  domain policy/validation logic lives under `interface/`. Review enforces.
- **R2 SENT-transition bypass / new send path** (approval_gate / state_ownership).
  Mitigation: single send entry = `ApproveEmailUseCase`; no direct `DeliveryService`
  call from any route. L4 checks the call graph.
- **R3 Secret leakage via /health, logs, or errors** (secrets_hygiene).
  Mitigation: central masking util used by logger + error envelope; `/health` returns
  only `status`+`version` from a single `__version__` constant; test asserts no
  sensitive key appears in any response/log fixture.
- **R4 Version drift** (multiple "0.1.0" literals). Mitigation: single `__version__`
  constant (decision for the Design Review: locate it — proposed `email_agent/__init__.py`).
- **R5 Over-building observability** (scope creep into metrics/tracing).
  Mitigation: OUT-of-scope list is explicit; any metrics/OTel item is rejected at
  design review.
- **R6 Coverage regression on inner layers** from refactor. Mitigation: M7 only adds
  outer layer + DI wiring; inner modules compile-tested unchanged; full pytest must
  stay green with no inner-layer edit.

## 9 · Design Review — RESOLVED (KICKOFF-INTERVIEW style, one Q at a time, 2026-07-17)

All open questions answered and locked. No code written; awaiting architecture approval
before any L1 BUILD.

### 9.1 User actions (Q1) — interface models user intent, not internal concepts
Exposed user actions (4): **Draft, Edit, Send, Reject**.
- "approve" is NOT a primary user-facing action. Send IS the approval+send action.
- Review is dropped as a server endpoint (see 9.3).
Delegation (AC-UI-1, one endpoint → one use case):
- **Draft**   → `DraftEmailUseCase.execute(DraftRequest)` → `DraftResult`.
- **Edit**    → `ApproveEmailUseCase.execute(draft, ApprovalDecision(decision="edit", …))`.
- **Send**    → interface delegates to exactly ONE application use case:
              `SendEmailUseCase.execute(draft, ApprovalDecision(decision="approve", approver=…))`.
              `SendEmailUseCase` (new, M7 — Option γ) is the single application-layer owner
              of the approve-then-deliver composition: it invokes `ApproveEmailUseCase`
              (M4, returns the frozen `EmailMessage`) and then `DeliveryService.send(message)`
              (M6) to deliver that EXACT instance. Per M6-D3 the caller (`SendEmailUseCase`)
              owns the SENT transition from the `DeliveryResult`. `ApproveEmailUseCase` and
              `DeliveryService` stay frozen and unchanged (Option γ, approved 2026-07-17).
              The interface only translates the unified `SendResult` — it does NOT compose
              the two use cases itself (that logic moved into the application layer).
- **Reject**  → `ApproveEmailUseCase.execute(draft, ApprovalDecision(decision="reject"))`.
- Regenerate is NOT a separate action — it is just another `Draft` call (same/updated request).

### 9.2 State model (Q2) — stateless, client-held workflow state
- M7 introduces NO server-side draft repository, cache, or in-memory workflow store.
- After `DraftEmailUseCase`, the interface translates `DraftResult` → a **`DraftResponseDTO`**
  carrying everything Edit/Send/Reject need (recipients/subject/body/tone/purpose/state/
  clarification payload). Client treats it as opaque workflow state and submits it back.
- Server RECONSTRUCTS the `EmailDraft` from the DTO, then calls exactly one use case
  (translation only — no business logic duplicated).
- No server-side lifecycle management; persistence explicitly deferred to a future milestone.

### 9.3 Review endpoint (Q3) — client/UI concern, not a server capability
- Option A1 chosen. **No Review endpoint.**
- The `DraftResponseDTO` returned by Draft IS the review payload; the client/UI renders it.
- A future GET (if persistence/draft-ids arrive) is intentionally out of M7 scope.

### 9.4 Approver identity (Q4 + Q5) — interface-owned resolution, B3 strict precedence
- Resolution lives ENTIRELY in the interface layer (a small resolver helper); the Application
  layer stays agnostic — it always receives a fully-populated `ApprovalDecision` and never
  inspects config/request context.
- Order: (1) request-supplied `approver` → (2) `settings.user.name` → (3) if neither, FAIL
  validation before invoking `ApproveEmailUseCase`.
- Backing config (Q5, C1): add minimal typed **`UserSettings`** section to `Settings`,
  `name: str = ""`, composed as `settings.user`. No signature/title/email/preferences.
  Follows the existing sectioned+validated `AnthropicSettings`/`GmailSettings` pattern.
  `app_name` is NOT reused (it is the product, not a human). No standalone env var.

### 9.5 Error envelope (Q6) — minimal, interface-owned, D1
```
{ "code": "<machine code>", "message": "<safe human-readable>", "detail": {…optional…} }
```
- Interface owns translation from Application results → envelope; App/Domain unaware of it.
- `code` examples: validation_error, clarification_required (normal branch, HTTP 200),
  approval_error, auth_required (DeliveryResult auth_failed → HTTP 401),
  delivery_unknown_state (unknown_state → verify-before-retry), delivery_failed
  (error/not_approved → HTTP 502), internal_error (HTTP 500).
- Messages NEVER contain secrets, prompts, bodies, OAuth tokens, or stack traces.
- `detail` is optional + machine-readable only.
- Un-typed exceptions → generic `internal_error`; full exception to structured logs ONLY.
- NO `request_id` (belongs to a future tracing milestone). NO `retryable` flag (would
  duplicate typed result semantics).

### 9.6 Locked at BUILD (these were the "still-open" items; all resolved by architecture approval)

- HTTP routes: `POST /draft`, `POST /edit`, `POST /send`, `POST /reject`, `GET /health`.
  No /review, /approve, /regenerate (decision #1).
- Version source: `email_agent.__version__` single constant; `AppSettings.app_version`
  mirrors it; `/health` returns `{status, version}` (decision #2).
- CLI: `argparse` (stdlib, zero runtime dep) (decision #3). No click/typer.
- DTOs: pydantic; domain never exposed; interface owns all translation (decision #4).
- Logging: stdlib `logging`, structured key/value, masks recipients/prompts/bodies/
  tokens/credentials; no JSON/tracing libs (decision #5).
- DI: app factory builds the graph once at startup; handlers receive constructed
  services; no infra inside handlers (decision #6).
- Error envelope: `{code, message, detail?}`, interface-owned; app/domain unaware
  (decision #7).
- Translation Boundary Principle recorded as ADR-TB (decision #8).

## 13 · M7 SEND COMPOSITION — Option γ APPLIED (post-build refinement, 2026-07-17)

The initial L1 BUILD (§11) had the interface compose `ApproveEmailUseCase` + `DeliveryService`
directly (Option α). Per explicit architecture directive, this was refactored to **Option γ**:
a new application-layer `SendEmailUseCase` now owns the approve-then-deliver composition.

- New file: `src/email_agent/application/send_email_use_case.py`
  (`SendEmailUseCase` + `SendResult`). It accepts `(EmailDraft, ApprovalDecision)`, invokes
  `ApproveEmailUseCase`, and — on APPROVED — forwards the EXACT `EmailMessage` to
  `DeliveryService.send`, returning a unified `SendResult`.
- Interface `/send` (API + CLI) now delegates to exactly ONE use case
  (`SendEmailUseCase`); it no longer composes the two use cases. AC-UI-1 preserved.
- `Container` builds `SendEmailUseCase` at startup (decision #6).
- `interface/errors.py` gained `send_result_error(SendResult)` mapping the unified result to
  the typed envelope (AuthRequiredError / DeliveryUnknownStateError / ApprovalError /
  DeliveryFailedError).
- Context-graph.json reconciled: stale edge `ApproveEmailUseCase → DeliveryService` removed
  (the VERIFIED M4 module returns EmailMessage and stops); new node `SendEmailUseCase` with
  edges to `ApproveEmailUseCase` + `DeliveryService`; `approval_gate` invariant updated.
- M4 (`ApproveEmailUseCase`) + M6 (`DeliveryService`) modules remain byte-for-byte unchanged
  (`git diff` empty). M4-E1 / M4-E4 / M6-X1 (artifact_identity) all preserved: the exact
  EmailMessage instance from approval is delivered.

Invariants confirmed by this change: AC-UI-1, artifact_identity (exact instance forwarded),
approval single-boundary (SendEmailUseCase is the only send path), M4/M6 freeze, separation
of concerns (composition in application layer, not transport), thin interface.
